## 2. Operating interpretation (non-normative)

This section is **not** part of the Resolution text. It records operating guidance for execution.

### 2.1 Mode transition summary

| Mode | Audience | What the audience sees | What is invisible |
|---|---|---|---|
| **Platform Mode** (PCR-001, PCR-002, ADR-001 era) | Platform operators / 사장님 | Engines, Standards, Playbooks, Knowledge Governance, Freeze Manifest, Recoveries | The end-customer value generated |
| **Client Project Mode** (PCR-003 era) | The client (e.g., Envoy Hostel & Tours) | Beautiful design, fast, bookable, manageable, SEO, mobile, AI chatbot | All Platform internals (Engines, Standards, Playbooks, etc.) |

### 2.2 Critical insight (paraphrased operational guidance)

> "지금까지는 Platform 중심으로 생각했습니다. 하지만 실제 사업에서는 고객 중심이어야 합니다. 즉 여러분의 플랫폼은 보이지 않는 운영체제입니다."

Operationally:
- The Platform internally uses every Engine, Standard, Playbook, QES gate.
- The client never sees a PRD, TRD, Agency OS, or Platform internals.
- The Platform performs Discovery → TRD → QES → Delivery as a single client-facing operation.
- The client only sees: the discovery conversation, the design, the deliverable, and the support.

### 2.3 What changes vs. PCR-002 era

PCR-002 established Product Lab 001 (PVP-001) as **the canonical Platform Validation Project** — running under Discovery Beta, with hypothesis-driven experiments to validate Platform assets.

PCR-003 elevates this to **Client Project Mode** — the same Platform internals, but now framed as **delivery to a paying client** (the client = Envoy Hostel & Tours). Specifically:

| PCR-002 (Product Lab) | PCR-003 (Client Project) |
|---|---|
| Audience: 사장님 + PVC | Audience: Envoy Hostel & Tours (the client) + 사장님 |
| Objective: validate Platform | Objective: deliver value to client |
| Mode of operation: experimental | Mode of operation: delivery |
| Output: validated hypotheses | Output: a production website |
| Pricing implication: Platform R&D cost | Pricing: **US$10,000+ agency premium** |

In practical terms: **PVP-001's existing deliverables become the internal working documents** of a Client Project, while a new Client-facing layer (the Brief, the Discovery Conversation, the Design Comps, the Delivery Checklist) becomes the externally-visible artifact.

### 2.5 Hidden Platform Principle (verbatim from PCR-003 §"OUTPUT")

> "플랫폼은 보이지 않는다. 고객의 결과물만 보인다."

Operationally:
- **The client sees only the deliverable**: a beautiful design, fast speed, bookings that work, easy CMS, AI chatbot, SEO, admin pages, mobile experience.
- **The Platform internals are hidden**: the client does not know there are 33 Platform Engines, ADR processes, or PVC reviews.
- **The Agency (Platform acting as agency) executes everything end-to-end automatically**: Customer Discovery → PRD → TRD → UI/UX → Frontend → Backend → CMS → SEO → QA → QES → Release.

### 2.6 Target value: US$10,000+

PCR-003 §"MISSION" specifies: "Deliver a production-ready premium agency-grade website. Target value US$10,000+".

Operationally this means:
- The deliverable is positioned at the **premium agency tier** (Kinfolk/Aman-grade visual quality — see `visual-design-kickoff.md` §1 for full design 헌장).
- The pricing/positioning document (US$10K+, ₾18 Lari value proposition) is filed as a Client-side artifact.
- The Brand DNA, Customer Discovery, Trust Evidence, Content Strategy, and Visual Design all align with premium-agency positioning.

## 3. Client Project structure

A Client Project consists of:

1. **Client-Facing Layer** (NEW, this Resolution creates):
   - Client Brief (the customer-visible specification)
   - Discovery Conversation record (client ↔ Platform dialogue)
   - Design Comps / Mockups (visible to client)
   - Client Review Checkpoints (4 checkpoints)
   - Client Acceptance Form

2. **Platform Internal Layer** (PRESERVED from PCR-002):
   - Pre-Coding Plan (gate-enforced)
   - 14-phase pipeline (renamed Client Project phases)
   - Agent Role Matrix (now operating in Client-facing mode)
   - All Phase deliverables (Discovery → Release)
   - QES gates (invisible to client)
   - EPS evidence collection (informs Platform evolution)

3. **Cross-Mapping**: each Client-facing artifact links to specific Platform-internal artifacts (without exposing internals to the client).
